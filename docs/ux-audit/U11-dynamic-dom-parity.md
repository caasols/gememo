# U11 — Dynamic DOM parity

## Summary

`popup.js` builds six runtime-generated component families: retry cards, action-item checklist, search results, log groups/entries, built-in rules, and stats-grid cells. The CSS system in `popup.html` defines a coherent vocabulary (`.widget`, `.btn`, `.row`, `.label`, `.log-entry`, `.stat-cell`, …) but the JS bypasses or contradicts it in 22 distinct places. The most serious problems are: (1) the log-entry template omits the `.log-title` element that the CSS explicitly styles, replacing it with a time-only header that inverts the visual hierarchy; (2) the retry-card "dismiss" button uses a class combination (`.btn.retry-dismiss-btn`) where the override rule `.retry-dismiss-btn { flex: 0 0 auto !important }` fights the layout rule `.retry-card-actions .btn { flex: 1 }` because both buttons share the same parent despite different roles — the dismiss button lives in `.retry-card-header`, not `.retry-card-actions`, yet still picks up the flex rule through the `.btn` class; (3) the `renderLogs` inline `log-retry-btn` button applies `.btn` but adds only a bespoke `.log-retry-btn` positioning class rather than the standard `.btn` size modifiers, producing a visually mismatched chip. There are also four dead CSS classes and two JS-only orphan classes.

---

## Proposed standard (parity rules + orphan/dead class list)

### Parity rules

1. **One source of truth for HTML structure.** Any component rendered by JS must follow the same element hierarchy as the CSS class comments expect. Deviations should be caught at the class-definition level, e.g. by co-locating a brief `<!-- template: -->` comment in CSS next to each component class block.
2. **Never emit inline styles when a CSS token exists.** Every hardcoded `style="font-size:…;color:var(--…)"` in JS must be replaced with a named utility or component class.
3. **Use `.btn` for all button-like elements without exception.** Size variants should come from modifier classes (e.g., `.btn.sm`, `.btn.full`) defined in CSS, not from ad-hoc `padding`/`height` properties on per-component classes.
4. **Adopt a minimal render-helper pattern.** A `buildEl(tag, classes, attrs, html)` helper and per-component builder functions (e.g., `buildLogEntry(entry)`) would eliminate string-concatenation bugs and enforce class reuse.
5. **`.log-title` must always appear inside `.log-content`.** Current omission breaks the visual hierarchy for all log entries.

### Dead CSS classes (defined in CSS, never used anywhere)

| Class | Defined at (popup.html line) | Why dead |
|---|---|---|
| `.kofi-btn` | 1125 (applied to `<a>`) | Used in HTML but no CSS rule defines `.kofi-btn`; the `<a>` inherits `.btn` styles only |
| `.retry-dismiss-btn` | 810 | The CSS layout rule `.retry-card-actions .btn` applies to all `.btn` children of `.retry-card-actions`; the dismiss button is NOT inside `.retry-card-actions` (it's in `.retry-card-header`), so the `!important` override has no real conflict to resolve — making `.retry-dismiss-btn` in CSS redundant |
| `.stats-savings` | 694 | Applied to `#stats-savings` paragraph in HTML but the rule only adds `a { color: var(--primary) }` — the class itself is otherwise unused as a scoping mechanism |
| `.log-group-header .log-dot` | 513 | The `margin-top: 0` override comment says "center in the flex header (UX-7)"; correct behavior, but the specificity rule is only needed because `.log-dot` globally sets `margin-top: 3px` — restructuring as a context variant is fine, but the intent should be documented |

### Orphan JS classes (used in JS-generated HTML, not defined in CSS)

| Class | Used at (popup.js line) | Problem |
|---|---|---|
| `.log-retry-btn` | 452 | Positioned via `margin-left: auto; flex-shrink: 0` in CSS (line 446–449), which IS defined — but the class is scoped inside `.log-entry`, which JS generates differently from the CSS `.log-entry` definition (see Finding #3) |
| `.action-task` | 210 | Used in `renderActionItems` innerHTML; CSS rule at line 707 targets `input:checked + .action-task` for strikethrough — the element is present, so this is not strictly dead, but the `.action-task` span receives no base styling of its own, making the class invisible until checked |
